# Gradle

**Gradle** is the second of Java's two dominant build tools, and the younger one. Where Maven configures builds in XML and locks us into a fixed lifecycle, Gradle describes builds in actual code — a **build script** written in Groovy or Kotlin — running on a flexible graph of tasks. The result is a tool that follows the same conventions as Maven by default but lets us customize anything when we need to. It's also the required build tool for Android, so it's worth knowing even for teams that are otherwise Maven shops.

---

## Project Structure

Gradle deliberately adopts Maven's directory conventions, so a Gradle project looks immediately familiar:

```
my-app/
├── build.gradle             # the build script
├── settings.gradle          # project name + subprojects
├── gradle/
│   └── wrapper/             # the Gradle Wrapper files (committed!)
├── gradlew                  # wrapper script (Linux/macOS)
├── gradlew.bat              # wrapper script (Windows)
├── src/
│   ├── main/
│   │   ├── java/            # application source code
│   │   └── resources/
│   └── test/
│       ├── java/            # test source code
│       └── resources/
└── build/                   # build output (Gradle's version of target/)
```

Same `src/main/java`, same `src/test/java` — the difference is that in Gradle these are *defaults we can change*, not assumptions baked into the tool. Build output goes to `build/` instead of `target/`. The `settings.gradle` file holds the project name and, in multi-module builds, lists the subprojects.

## The Build Script: `build.gradle`

Instead of a POM, Gradle uses a build script — by default `build.gradle`, written in the **Groovy DSL**. A complete, working build file for a Java application:

```groovy
plugins {
    id 'java'
    id 'application'
}

group = 'com.example'
version = '1.0.0-SNAPSHOT'

java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(21)
    }
}

repositories {
    mavenCentral()
}

dependencies {
    implementation 'com.fasterxml.jackson.core:jackson-databind:2.17.1'
    testImplementation 'org.junit.jupiter:junit-jupiter:5.10.2'
    testRuntimeOnly 'org.junit.platform:junit-platform-launcher'
}

application {
    mainClass = 'com.example.App'
}

test {
    useJUnitPlatform()
}
```

Notice how much shorter this is than the equivalent `pom.xml`. The `plugins` block does the heavy lifting: applying the `java` plugin gives us compile, test, and jar tasks plus the standard directory conventions, all in one line.

One note before we go further: Gradle also supports a **Kotlin DSL** — the same script written in Kotlin, in a file named `build.gradle.kts`. It's now the default for new projects generated by `gradle init`, and the concepts are identical; only small syntax details differ (double quotes, parentheses on function calls). Everything in these notes applies to both. We'll use the Groovy DSL here since it's what most existing codebases contain.

### Declaring Repositories and Dependencies

The `repositories` block says where to look — `mavenCentral()` covers most needs, and entries are searched in order:

```groovy
repositories {
    mavenCentral()
    maven { url = 'https://repo.example.com/private' }   // a private repo
}
```

The `dependencies` block says what we need, using the familiar `group:artifact:version` coordinates as a single string. Instead of Maven's scopes, Gradle uses **configurations**:

| Gradle configuration | Closest Maven scope | Meaning |
|---|---|---|
| `implementation` | `compile` | Normal dependency; *not* exposed to consumers of our library |
| `api` | `compile` | Like `implementation`, but exposed to consumers (library projects only) |
| `testImplementation` | `test` | Available only to test code |
| `compileOnly` | `provided` | Compile-time only; not packaged, not at runtime |
| `runtimeOnly` | `runtime` | Runtime only; not on the compile classpath (e.g., JDBC drivers) |

The `implementation` vs `api` distinction is a genuine improvement over Maven: `implementation` dependencies stay hidden from downstream projects, which keeps compile classpaths smaller and rebuilds faster. Default to `implementation`; reach for `api` only when our library's public methods expose the dependency's types.

### Tasks vs Lifecycle Phases

This is the biggest conceptual shift from Maven. Maven has a fixed lifecycle — a straight line of phases. Gradle instead builds a **directed graph of tasks**. A **task** is a single unit of work (`compileJava`, `test`, `jar`), and tasks declare dependencies on each other. Asking for a task runs everything it depends on, in order. The familiar `build` task is just a top-of-graph task that depends on the rest:

```
compileJava → processResources → classes → jar
classes → compileTestJava → testClasses → test
jar + test → check → build
```

Two practical consequences:

- **Incremental builds.** Each task knows its inputs and outputs. If nothing a task depends on has changed since the last run, Gradle skips it and reports it as `UP-TO-DATE`. Large projects rebuild in seconds because most tasks don't actually run.
- **Custom tasks are first-class.** Since the build script is code, defining a new task is trivial:

```groovy
tasks.register('printVersion') {
    doLast {
        println "Building version $version"
    }
}
```

There's no Maven-style ceremony of binding plugin goals to lifecycle phases — we write a task, wire it into the graph if needed, and run it by name.

## The Gradle Wrapper

The **Gradle Wrapper** is a small script — `gradlew` (and `gradlew.bat` for Windows) — committed into the repository alongside the code. Running `./gradlew build` instead of `gradle build` makes the wrapper download and use the *exact Gradle version the project specifies* (pinned in `gradle/wrapper/gradle-wrapper.properties`), rather than whatever happens to be installed on the machine.

This matters more than it first appears:

- **Nobody installs Gradle.** A fresh machine with just a JDK can clone the repo and run `./gradlew build` — the wrapper bootstraps everything.
- **Everyone builds with the same version.** Gradle versions differ meaningfully; the wrapper eliminates "works on my machine" problems caused by version drift, on laptops and CI alike.
- **Upgrades are deliberate.** Bumping the Gradle version is a one-line change in the wrapper properties, reviewed like any other code change.

The rule of thumb on any real team: **always invoke the wrapper, never a globally installed `gradle`**. If a project has a `gradlew` file, use it.

## Common Commands

Everyday Gradle, via the wrapper (on Windows, `gradlew.bat` or just `.\gradlew`):

```bash
./gradlew build           # compile, test, and produce the jar in build/libs/
./gradlew test            # compile and run the tests
./gradlew clean           # delete the build/ directory
./gradlew clean build     # full fresh build
./gradlew run             # run the app (needs the 'application' plugin)
./gradlew tasks           # list all available tasks — great for discovery
./gradlew dependencies    # print the dependency graph (Maven's dependency:tree)
```

A few flags worth knowing:

```bash
./gradlew build -x test       # build but exclude the test task (use sparingly)
./gradlew test --tests OrderServiceTest    # run a single test class
./gradlew build --info        # more detail; --debug for much more
```

One behavior that surprises Maven users: thanks to incremental builds, a second `./gradlew build` with no changes finishes almost instantly with most tasks marked `UP-TO-DATE`. That's not the build being skipped incorrectly — that's the task graph doing its job. When we genuinely want everything re-done from scratch, `clean build` forces it.
