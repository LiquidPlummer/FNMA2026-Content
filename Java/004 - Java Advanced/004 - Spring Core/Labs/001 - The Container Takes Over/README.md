# Lab: The Container Takes Over — IoC, DI, Beans & the Application Context

In this lab we build a small greeting/notification app **twice**: first wired by hand in `main` (so the pain of manual dependency management is real, not theoretical), then handed to Spring — `@SpringBootApplication`, stereotype annotations, constructor injection, `@Qualifier`, `@Primary`, a `@Bean` method for a class we can't annotate, and `List` injection to fan out across every sender in the container. Along the way we'll provoke the two classic wiring failures on purpose and learn to read their startup errors.

No web layer yet — the app runs and exits via `CommandLineRunner`. The container is the lesson.

---


By the end of the lab that same command runs a fully wired app.

## Part 1 — The manual version: feel the pain first

Before Spring earns its keep, we need the problem. We'll build the domain with plain classes — no annotations anywhere — and wire it ourselves.

Create four files in `src/main/java/com/curriculum/labs/`:

**`NotificationSender.java`** — the abstraction everything hangs on:

```java
package com.curriculum.labs;

public interface NotificationSender {
    void send(String message);
}
```

**`EmailSender.java`** and **`SmsSender.java`** — two implementations:

```java
package com.curriculum.labs;

public class EmailSender implements NotificationSender {
    @Override
    public void send(String message) {
        System.out.println("[email] " + message);
    }
}
```

```java
package com.curriculum.labs;

public class SmsSender implements NotificationSender {
    @Override
    public void send(String message) {
        System.out.println("[sms]   " + message);
    }
}
```

**`GreetingService.java`** — the business logic, depending only on the interface:

```java
package com.curriculum.labs;

public class GreetingService {

    private final NotificationSender sender;

    public GreetingService(NotificationSender sender) {
        this.sender = sender;
    }

    public void greet(String name) {
        sender.send("Welcome aboard, " + name + "!");
    }
}
```

Now the wiring. Create **`ManualWiring.java`** — an ordinary `main`, no Spring:

```java
package com.curriculum.labs;

public class ManualWiring {

    public static void main(String[] args) {
        NotificationSender email = new EmailSender();
        NotificationSender sms = new SmsSender();
        GreetingService service = new GreetingService(email);

        service.greet("Ada");
    }
}
```

*Every object built by hand, in dependency order, by the caller.*

Run it (no pom changes needed — we invoke the exec plugin directly):

```console
mvn -q compile org.codehaus.mojo:exec-maven-plugin:3.5.0:java -Dexec.mainClass=com.curriculum.labs.ManualWiring
```

`[email] Welcome aboard, Ada!` — it works, and for four classes it's honestly fine. Now project forward: every new service means another line here, *in the right order*; every constructor that grows a parameter breaks this file; swapping email for SMS app-wide means hunting down construction sites; and `main` has become the one place that knows every class in the system. This file is the job Spring is about to take. **Inversion of control** means exactly this handoff: objects stop being constructed by their callers and start being *provided* — the container news up the objects and hands them their dependencies, instead of our code doing it. Dependency injection is the mechanism (dependencies arrive through the constructor); the container is the provider.

## Part 2 — Spring takes over (and the first failure mode)

Spring discovers classes by **component scanning**: at startup it walks the classpath under the application's package, finds classes wearing stereotype annotations, constructs one instance of each — a **bean** — and stores it in the **application context**, injecting constructor arguments from the other beans it knows.

Step one, annotate the *service* — add `@Service` above `GreetingService`'s class declaration (import `org.springframework.stereotype.Service`):

```java
@Service
public class GreetingService {
```

*A stereotype: "container, this class is yours — construct it, manage it, inject it."*

Step two, make the application ask for it. Replace the body of `GreeterApplication` with:

```java
@SpringBootApplication
public class GreeterApplication implements CommandLineRunner {

    private final GreetingService service;

    public GreeterApplication(GreetingService service) {
        this.service = service;
    }

    public static void main(String[] args) {
        SpringApplication.run(GreeterApplication.class, args);
    }

    @Override
    public void run(String... args) {
        service.greet("Ada");
    }
}
```

*`CommandLineRunner`: the container starts, wires everything, then calls our `run` — a console app with a fully managed object graph.*

Note the shape of the injection: a **`final` field, set in the constructor** — no setter, no field annotation. Constructor injection is the standard for good reasons the compiler enforces: the bean can't exist half-wired, and `final` documents that dependencies never change.

Now — deliberately — do **not** annotate the senders yet. Run `mvn spring-boot:run` and read the failure:

```
Parameter 0 of constructor in com.curriculum.labs.GreetingService required a bean
of type 'com.curriculum.labs.NotificationSender' that could not be found.
Action: Consider defining a bean of type 'com.curriculum.labs.NotificationSender' ...
```

**Zero candidates.** The container needed a `NotificationSender` and its scan found none — because plain classes without stereotypes are invisible to it. This is the single most common Spring error you'll ever see, and its everyday causes are a missing annotation (our case, right now) or a class outside the scanned packages (exercise 3's case). Fix it: add `@Component` to `EmailSender` (import `org.springframework.stereotype.Component`), run again — `[email] Welcome aboard, Ada!`, this time with nobody constructing anything by hand.

## Part 3 — Two candidates: @Qualifier and @Primary

Add `@Component` to `SmsSender` as well, run again, and meet failure mode number two:

```
Parameter 0 of constructor in com.curriculum.labs.GreetingService required a single bean,
but 2 were found: emailSender ... smsSender ...
```

**Two candidates.** The container found *two* beans implementing `NotificationSender` and refuses to guess. (Note the bean names in the message: by default a component's name is its class name with a lowercase first letter.) Two tools resolve the ambiguity, from opposite directions:

**`@Qualifier` — the injection point chooses.** In `GreetingService`:

```java
public GreetingService(@Qualifier("emailSender") NotificationSender sender) {
    this.sender = sender;
}
```

*"Of the candidates, give me the one named `emailSender`."*

(Import `org.springframework.beans.factory.annotation.Qualifier`.) Run — working again. **`@Primary` — one candidate declares itself the default.** Try the alternative: remove the `@Qualifier`, and instead put `@Primary` (import `org.springframework.context.annotation.Primary`) on `EmailSender` next to its `@Component`. Run — same result, different philosophy: `@Primary` sets a system-wide default that any injection point can still override with a `@Qualifier`. Keep the `@Primary` version; we'll rely on it from here.

## Part 4 — The class we can't annotate: @Bean

Look at [TemplateEngine.java](src/main/java/com/curriculum/labs/TemplateEngine.java) — provided, and by the rules of this lab it's a *third-party library class*: we can't add `@Component` to code we don't own. Let's create the need first. Upgrade `EmailSender` to format through it:

```java
@Component
@Primary
public class EmailSender implements NotificationSender {

    private final TemplateEngine templates;

    public EmailSender(TemplateEngine templates) {
        this.templates = templates;
    }

    @Override
    public void send(String message) {
        System.out.println("[email] " + templates.render(message));
    }
}
```

Run: the zero-candidates error again — this time for `TemplateEngine`, and this time an annotation is off the table. The answer is a **`@Bean` method**: a factory method we write, whose *return value* the container manages. Add it to `GreeterApplication` (imports: `org.springframework.context.annotation.Bean`):

```java
    @Bean
    TemplateEngine templateEngine() {
        return new TemplateEngine("** ");
    }
```

*Component scanning finds classes; `@Bean` methods hand the container objects we constructed ourselves — the escape hatch for library types.*

Run: `[email] ** Welcome aboard, Ada!`. Same container, two registration styles — `@Component` for our classes, `@Bean` for everyone else's. (This is why `@SpringBootApplication` implies `@Configuration`: the application class is allowed to host `@Bean` methods.)

## Part 5 — All of them at once: List injection

The container knows every `NotificationSender` bean. Ask for the whole set: add a broadcast path to `GreetingService`:

```java
@Service
public class GreetingService {

    private final NotificationSender sender;
    private final List<NotificationSender> allSenders;

    public GreetingService(NotificationSender sender, List<NotificationSender> allSenders) {
        this.sender = sender;
        this.allSenders = allSenders;
    }

    public void greet(String name) {
        sender.send("Welcome aboard, " + name + "!");
    }

    public void announce(String message) {
        for (NotificationSender s : allSenders) {
            s.send("[ANNOUNCEMENT] " + message);
        }
    }
}
```

*Injecting `List<NotificationSender>`: the container collects every matching bean into the list.*

Add `service.announce("Cutover is Saturday 02:00");` to `run(...)` in `GreeterApplication` and run: the greeting goes to email (the `@Primary`), the announcement hits **both** senders. The detail that makes this pattern quietly powerful: when a new sender joins the app, `announce` picks it up with *zero* edits to this class — which exercise 1 will prove.

## Exercises

The training wheels come off — no step-by-step from here.

1. **The third sender.** Add a `SlackSender` (prints `[slack] ...`). First run the app changing nothing else and check the announcement fan-out — how many edits did the broadcast need? Then make the individual `greet` go to Slack instead of email, in the way that touches the *fewest* files — and write one sentence on when you'd reach for `@Qualifier` at the injection point versus moving `@Primary`.

2. **The cache warmer.** Create a `GreetingHistory` bean that keeps a list of past greetings, and give it a method annotated `@PostConstruct` (import from `jakarta.annotation`) that pre-loads two entries and prints a line when it runs. Inject it into `GreetingService` and record every greeting. Run and study the log order: constructor first, `@PostConstruct` after — then verify that claim by printing from `GreetingHistory`'s constructor too. Answer in a comment: why can't the pre-load happen in the constructor of a bean that (hypothetically) needed another bean to compute its entries?

3. **The component-scan mystery.** Create a new package `com.curriculum.other` (note: **not** under `com.curriculum.labs`) containing a `PagerSender` implementing `NotificationSender`, annotated `@Component`. Run the app. The announcement doesn't include it, no error, nothing — the bean simply doesn't exist. Explain why from what `@SpringBootApplication` implies about *where scanning starts*, then fix it **without moving the class** (there's an attribute for exactly this) — and then state why moving the class is nevertheless the better fix.
