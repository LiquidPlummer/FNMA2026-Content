# Preventing Blind-Acceptance

The most dangerous property of AI-generated code is that it *looks right*. It compiles, it's idiomatic, it's confidently formatted, and it arrives faster than scrutiny can keep up. **Blind acceptance** — Tab, Tab, commit — is the failure mode that turns a productivity tool into a defect pipeline, and the research on automation is blunt: fluent output systematically lowers human vigilance exactly when it shouldn't. This lesson is the countermeasure kit.

---

## Why Plausible ≠ Correct

An LLM generates the most *statistically likely* code for the prompt — which overlaps with "correct" heavily but not reliably. The gaps have recognizable shapes:

- **The typical-case solution** — handles the happy path the way most code in training data does; misses *our* edge case (the partial refund, the null customer, the leap year).
- **The subtly wrong constant/operator** — `>=` for `>`, `plusDays(30)` for a month, the wrong rounding mode. Reads fine at a glance; fails in reconciliation.
- **The invented API** — a method that *should* exist and doesn't (hallucination — dedicated lesson later). At least this one fails loudly at compile time.
- **The quiet scope creep** — the diff we asked for, plus an "improvement" we didn't: a renamed variable, a reformatted block, a swallowed exception. Diffs must be read *whole*.
- **Confidently wrong explanations** — the prose *about* the code is generated by the same plausibility engine as the code. "This ensures thread safety" is a claim, not a fact.

None of these announce themselves. All of them pass a skim.

---

## The Review Disciplines

**The junior-dev standard.** Review AI output exactly as we'd review a PR from a talented new hire with no knowledge of our domain: capable, fast, zero accountability. Concretely — read every line before accepting; for multi-file changes, review the *diff*, not the chat summary of it. If reviewing would take longer than writing, the delegation was wrong-sized: re-scope smaller.

**Verification is mechanical, not vibes.** The referee is never the AI's confidence — it's the compiler, the test suite, and execution:

```text
After any accepted change:
  1. Does it compile / does the linter pass?         (free)
  2. Do existing tests pass?                          (cheap)
  3. Is there a NEW test proving the new behavior?    (the real check —
     and write or review this test with extra care: an AI-written test
     for AI-written code can share the same misunderstanding)
  4. For logic we can't test easily: trace one concrete input through
     the code by hand. One worked example catches most subtle-operator bugs.
```

*The acceptance ladder — each rung catches what the previous one can't, and none of them involve trusting prose.*

**Interrogate before accepting.** Cheap prompts that surface problems: *"What edge cases does this not handle?"* — the AI is oddly good at critiquing its own draft when asked. *"What assumptions does this make about the inputs?"* *"Walk through what happens when the list is empty."* Asking costs seconds; the answers regularly reveal the gap we'd have found in production.

**The understanding bar.** The non-negotiable minimum: *we can explain every line we commit.* If a generated block uses a construct we can't explain, that's a lookup-and-learn moment (previous topic) before it's a commit. "It works but I don't know why" is not a state code ships in — that's comprehension debt with interest.

---

## Calibrated, Not Paranoid

Blanket distrust wastes the tool as surely as blanket trust misuses it. Scale scrutiny to blast radius: generated *test data* gets a skim; a generated *getter* gets a glance; generated *business logic* gets the full ladder; generated *security, money, or migration* code gets the ladder plus a human second reviewer. The skill being built is **calibration** — knowing which outputs have earned which level of trust, and never letting fluency stand in for evidence.

This discipline is the foundation everything after assumes — the techniques in the next unit make the AI *more* capable, which only raises the stakes on reviewing it well.
